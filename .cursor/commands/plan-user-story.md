# ユーザーストーリー作成（Plan モード）

まず Plan モードに切り替えてから作業を開始すること。
コード変更やファイル編集は承認されるまで一切行わない。

## 基本姿勢

ユーザーの指示を受動的に反映するだけでなく、**自ら能動的に思考する**こと。
ペルソナの立場に立ち、「この機能があればもっと便利になる」「このギャップを埋めれば価値が最大化する」という観点で、積極的にストーリーを提案・追記する。

## ワークフロー

### 1. 現状分析（能動的に実施）

`docs/user-stories-webhook-dashboard.md` を読み込み、以下を分析する:

- 現在の最大ストーリー ID（Phase 1-5 系: `US-0XX`、Phase 6 系: `US-1XX`）
- 各フェーズの構成と既存ストーリーの一覧・ステータス
- 「ギャップ」セクションに記載された未解決課題
- 「追加で検討すべきユーザーストーリー（提案）」セクションの内容
- ペルソナの典型的なワークフローで、まだカバーされていないステップ

### 2. ギャップ分析と能動的な提案

ユーザーの要望を聞く**前に**、以下の観点で自ら分析し、提案候補を洗い出す:

**ペルソナ視点の分析:**
- メインペルソナ（Web3エンジニア）の典型ワークフロー 6 ステップのうち、既存ストーリーでカバーできていない体験はないか
- サブペルソナ（QA / SRE / CS）の業務フローで不足している機能はないか
- 既存の【完了】ストーリーを使った後に「次にほしくなる」自然な体験は何か

**プロダクト視点の分析:**
- 既存フェーズ間の接続で抜けている導線はないか
- 「ギャップ」セクションに書かれているが、まだストーリー化されていない課題はないか
- 「追加で検討すべきユーザーストーリー（提案）」から正式にストーリー化すべきものはないか
- 競合ツール（webhook.site 等）にあってこのプロダクトにない機能はないか

### 3. ユーザーとの対話

AskQuestion ツールを使い、以下を行う:

- ユーザーが特定の要望を持っている場合はそれを聞く
- **加えて**、ステップ 2 で洗い出した提案候補を選択肢として提示し、ユーザーの関心を確認する
- 対象ペルソナ、フェーズ、優先度について合意を取る

ユーザーが既にチャット入力で概要を伝えている場合は、不足情報のみ質問しつつ、関連する追加ストーリーも能動的に提案する。

### 4. ストーリー起草

以下のフォーマットに従ってストーリーを作成する:

```
#### {ID} {タイトル}（{優先度}）【未着手】

{ペルソナ}として、{実現したいこと}したい。
なぜなら{理由}から。

- 受け入れ基準
  - Given {前提条件}、When {操作}、Then {期待結果}
  - Given {前提条件}、When {操作}、Then {期待結果}
```

ルール:
- ID は既存の最大 ID の次番号を採番する
- 受け入れ基準は最低 2 つ、正常系と異常系/境界系を含める
- ストーリーは「ユーザー価値」を中心に保ち、実装タスクに寄せすぎない
- ユーザーの要望に加え、分析で見つけた関連ストーリーもセットで起草する
- 「このストーリーがあると、既存の XX がさらに活きる」という相乗効果を意識する

### 5. プラン提示

CreatePlan ツールでストーリー案を提示する。プランには以下を含める:

- 起草したストーリー全文（ユーザー要望 + 能動提案を区別して記載）
- 挿入先（どのフェーズ・どのストーリーの後に追加するか）
- 既存ストーリーとの関連や依存
- なぜこのストーリーが価値を生むのか（ペルソナ視点の根拠）

### 6. 承認後の反映

ユーザーが承認したら Agent モードに切り替え、`docs/user-stories-webhook-dashboard.md` の該当フェーズセクションにストーリーを追記する。
