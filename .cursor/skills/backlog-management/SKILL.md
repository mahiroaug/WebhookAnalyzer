# バックログ管理

ユーザーストーリー文書を起点に、実装可能なバックログアイテムへ分解しスプリント計画を立てるスキル。

## バックログ切り出しワークフロー

1. プロジェクトのユーザーストーリー文書を読む
2. 対象ストーリーの受け入れ基準（Given/When/Then）を確認する
3. 実装タスクに分解する（フロント / バックエンド / テスト）
4. ストーリー間の依存関係を整理する
5. TodoWrite でタスクリストを作成する

## ストーリー → タスク分解テンプレート

```
{ストーリーID}: {ストーリータイトル}

バックエンド:
- [ ] API エンドポイント追加/修正
- [ ] スキーマ追加/修正
- [ ] DB モデル変更
- [ ] サービスロジック

フロントエンド:
- [ ] ページコンポーネント
- [ ] 共通コンポーネント
- [ ] API クライアント関数
- [ ] ルーティング追加

検証:
- [ ] 受け入れ基準 1: {Given/When/Then}
- [ ] 受け入れ基準 2: {Given/When/Then}
```

不要なレイヤーは省略してよい（例: DB変更なしならモデル行を削除）。

## 優先度の判断基準

| 優先度 | 基準 |
|--------|------|
| P0 | ないとプロダクトとして成立しない |
| P1 | あるとユーザー体験が大きく向上する |
| P2 | 将来の拡張やチーム運用で必要になる |

## スプリント計画の考え方

- 1スプリント = 1〜3 ストーリー（規模に応じて調整）
- P0 を優先し、同一フェーズ内は番号の若い順に着手する
- バックエンドAPI変更が必要なストーリーはAPI→UIの順で実装する
- 既存APIで成立するストーリーは即着手可能と判断する

## 依存関係の整理方法

ストーリー間の依存は以下の観点で判断する:

- **画面遷移**: 遷移元の画面が未実装なら、先にそちらを実装する
- **データ依存**: 表示に必要なAPIやDBスキーマが未実装なら先行する
- **UI部品**: 共通コンポーネントが必要なら先に切り出す

依存関係はユーザーストーリー文書内、またはタスクリストに記録する。

## ステータス管理

| ステータス | 意味 |
|-----------|------|
| 未着手 | バックログに存在するが未実装 |
| 実装中 | 開発作業中 |
| 受け入れ待ち | 実装完了、受け入れ基準の確認待ち |
| 完了 | 全受け入れ基準を満たした |
| 持ち越し | 未達基準があり次スプリントへ |

ステータスはユーザーストーリー文書に記録する。
