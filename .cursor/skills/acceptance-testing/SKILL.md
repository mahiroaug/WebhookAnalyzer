# 受け入れ検証

ユーザーストーリーの Given/When/Then 受け入れ基準に沿ってブラウザで動作確認し、結果を記録するスキル。

## 検証ワークフロー

```
検証チェックリスト:
- [ ] Step 1: 対象ストーリーの受け入れ基準を読む
- [ ] Step 2: 前提条件（Given）を準備する
- [ ] Step 3: 操作（When）をブラウザで実行する
- [ ] Step 4: 結果（Then）を確認・スクリーンショット取得する
- [ ] Step 5: 結果を記録する
```

## Step 1: 受け入れ基準の確認

プロジェクトのユーザーストーリー文書から、対象ストーリーの受け入れ基準を読み取る。

基準は Given/When/Then 形式で記述されている:
- **Given**: 前提条件（テストデータ・画面状態）
- **When**: ユーザー操作（クリック・入力・遷移）
- **Then**: 期待結果（表示内容・挙動）

## Step 2: 前提条件の準備

Given の状態を作るために必要な操作を行う。

- テストデータの投入（スクリプト or API呼び出し）
- 必要なサービスの起動確認
- 画面の初期状態を整える

## Step 3: ブラウザでの操作

- ブラウザツール（browser_navigate, browser_snapshot 等）で画面を操作する
- When に記述された操作を忠実に再現する
- 操作前後で snapshot を取得して状態変化を記録する

## Step 4: 結果の確認

Then に記載された期待結果と実際の画面状態を照合する。

確認観点:
- 表示内容が期待と一致するか
- UI要素が正しく描画されているか
- エラーが発生していないか（コンソール含む）
- レスポンス速度に著しい問題がないか

## Step 5: 結果の記録

検証結果を以下の形式で報告する。

```markdown
## {ストーリーID} 受け入れ検証結果

| # | 受け入れ基準 | 結果 | 備考 |
|---|-------------|------|------|
| 1 | Given ... When ... Then ... | OK / NG | 補足があれば |
| 2 | Given ... When ... Then ... | OK / NG | 補足があれば |

**判定**: 完了 / 持ち越し（未達基準: #X）
```

## 合否判定

- 全基準 OK → ストーリーを「完了」にする
- 1つでも NG → 「持ち越し」とし、NG 理由を記録する
- NG の場合は修正タスクを TodoWrite で追加する
