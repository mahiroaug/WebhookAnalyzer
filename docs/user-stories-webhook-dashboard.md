# Webhook UI ユーザーストーリー

## 目的

Webhook Analyzer を `webhook.site` のような「受信確認 + 高速調査 + AI解説」ができるUIに進化させる。  
まずは「ダッシュボードが実際に動く」状態をMVPとし、段階的に高機能化する。

最終的には、外部サービス（Fireblocks, BitGo, Alchemy 等）から届くWebhookの形式を**網羅的に調査**し、その結果を**調査レポートとして出力**できるツールを目指す。

## スコープ

- 対象: フロントエンドUI（一覧・詳細・分析・レポート体験）
- 関連: 既存API（一覧/詳細/統計/分析）と将来API（WebSocket、比較、共有、エクスポート）
- 非対象: 外部Webhook送信元の個別SDK実装

## 現状（2026-02時点）

- 一覧表示（`source` / `event_type` のフィルタ）
- 統計カード表示（source別 / event_type別）


### ギャップ

- 個別Webhook詳細表示の導線・実装不足
- 詳細画面のJSON要素分解（ツリー/型表示/コピー導線）不足
- AI分析実行と分析結果表示（summary / field_descriptions）未実装
- AI解説のUX（失敗時再試行、分析ステータス、根拠表示）不足
- WebSocketリアルタイム反映未実装
- event_type別グルーピングやスキーマ推定機能なし
- 調査レポートのエクスポート機能なし
- 一括分析・セッション管理機能なし

## ペルソナ

### メインペルソナ: Web3エンジニア（調査・レポート）

外部サービス（例: Fireblocks）の新機能や監査系Webhookを様々なパターンで送信し、**どんな形式の通知が届くかを網羅的に調査**したいエンジニア。調査結果は社内の技術ドキュメントや連携設計書として**レポートにまとめたい**。

典型的なワークフロー:
1. ngrok等で公開URLを取得し、外部サービスのWebhook送信先に設定する
2. 外部サービス上で様々な操作（送金、承認、アカウント変更等）を行い、Webhookを飛ばす
3. 受信した通知の一覧を確認し、event_type ごとに分類・整理する
4. 各Webhookの詳細を開き、payloadの構造・フィールドの意味をAIで解説させる
5. 同一event_typeの複数パターンを比較し、差分や共通構造を把握する
6. 調査結果をレポートとしてエクスポートし、チームに共有する

### サブペルソナ

1. QA（期待イベントの到達確認・比較検証）
2. SRE/運用担当（異常イベント監視・トリアージ）
3. CS/サポート（顧客事象の再現確認）

## フェーズ方針

- Phase 1（MVP）: 「受信状況を安全に追える一覧」を成立させる
- Phase 2（詳細強化）: 「1件のWebhookをすぐ理解できる詳細」へ強化
- Phase 3（AI高度化）: 「読み解きコストを下げるAI支援」を強化
- Phase 4（運用性拡張）: 「継続運用しやすいリアルタイム調査基盤」へ拡張
- Phase 5（調査・レポート）: 「網羅調査→レポート出力」までの一気通貫体験を実現する
- Phase 6（デザイン刷新）: PolygonScan DIM + Apple HIG ベースの洗練されたUIへ全面刷新する
- Phase 7（開発ツール UX 刷新）: webhook.site 風 2ペインレイアウト・Payload 表形式・HTTPメタデータ保存で開発ツールとしての実用性を根本から向上させる
- Phase 8（UI バグ修正と検索強化）: Phase 7 で発見されたバグの修正、source アイコン、全文検索、AI 分析堅牢化で品質を底上げする
- Phase 9（AI 分析結果の定義ファイル化）: AI 分析結果を YAML 定義ファイルとして永続化し、キャッシュ読み込みで LLM 不要の即時表示を実現する
- Phase 10（AI 分析品質とセキュリティ）: 3 層出力化・サニタイズ・フィードバック再分析・Web 検索で AI 分析の品質と安全性を強化する
- Phase 11（画面遷移リファインメントと分析可観測性）: ページ遷移の暗転除去、分析ボタンのスピナー、SSE による分析ログのリアルタイムストリーミングで操作体験と可観測性を向上させる
- Phase 12（DB 運用・メンテナンスツール）: DB 統計・検索・削除・分析リセット・メンテナンスの CLI ツールで運用作業を効率化する
- Phase 13（高度機能・開発者体験）: Replay・異常検知ルールなど高度な開発者向け機能を追加する
- Phase 14（AI 分析 UX 改善とバグ修正）: Payload テーブルの全展開/折りたたみ、分析ログのリアルタイムカウント・詳細拡充、未知バッジ即時解除、セクション配置最適化、Step 2 フィールド欠落バグ修正で AI 分析体験を総合的に改善する
- Phase 15（リアルタイム体験と分析ログの品質改善）: 分析ログの Webhook 切替時リセット・全タイムスタンプ付与、表示領域拡大、INBOX への受信 URL 表示、WebSocket リアルタイム更新修復、Payload 全展開の一括展開修正で日常利用の品質を底上げする
- Phase 16（表示体験の総合改善と PDF エクスポート）: Payload のテーブル/JSON 切替、全展開デフォルト化、未読管理、500 件表示、サービス接続状況可視化、event_type プルダウン、インクリメンタルサーチ、ボタン英語統一、PDF レポート出力で日常的な調査体験を総合的に向上させる
- Phase 17（表示品質改善・PDF 拡充・フィルタ連動）: ServiceStatusPanel の表形式化、PDF の Payload テーブル形式出力・メタデータ追加・AI 分析結果修正、未読ハイライト強化、日時書式統一、ナビバー位置調整、source/event_type フィルタ相互連動で表示品質と操作性を底上げする
- Phase 18（INBOX ヘッダー簡素化）: INBOX ヘッダーの構成を簡素化し URL 全文表示で視認性を向上させる
- Phase 19（サービスステータスの表示改善とヘルスチェック信頼性向上）: ServiceStatusPanel の Part/Component 2列化・絵文字ステータス・コンパクト表示と、レイテンシ計測・モデル可用性・エラー理由表示によるヘルスチェックの信頼性向上
- Phase 20（お気に入り管理）: 一覧ペインの各エントリに星アイコンでお気に入りマークを付与し、Favorites Only フィルタで注目エントリだけに絞り込めるようにする

## 実装時の注意

- ストーリーは「ユーザー価値」を中心に保ち、実装タスクに寄せすぎない
- 既存APIで成立する範囲（一覧/詳細/統計/分析）と追加開発が必要な範囲（WebSocket/比較/共有）を分離する
- 各ストーリーは将来Issue化しやすいように独立性を意識する
